## Разработка комплекса скриптов автоматизированного развертывания сервера, CMS, VPN и интеграции с AI (DeepSeek)

**Версия 11.1** – уточнённая: полные последовательности действий для VPN и AI, автоматизация внедрения трекера через админку, поддержка конвертации существующих сертификатов.

---

## 1. Общие положения

### 1.1. Цель проекта
Создать набор последовательных и модульных скриптов для автоматической установки и настройки сервера под управлением Linux, развертывания на нём **готовой инфраструктуры для сайта** (веб-сервер, SSL, база данных) вместе с **системой управления контентом (CMS)**, добавления VPN-сервера на том же домене и последующей интеграции модуля генерации контента на основе AI DeepSeek. Все скрипты должны выполняться на «чистой» системе с минимальным участием оператора и обеспечивать получение полностью работоспособного продукта.

### 1.2. Общая концепция и модульность («кубики»)
Проект состоит из **четырёх основных скриптов** и одной общей библиотеки:

1. **init.sh** – базовая подготовка сервера (обязательный для всех).
2. **cms.sh** – **объединённый скрипт**: установка CMS + настройка домена, веб-сервера (Nginx), SSL-сертификата, базы данных.
3. **vpn.sh** – развертывание VPN-сервера (3X-UI) с интеграцией на одном домене.
4. **ai.sh** – интеграция с AI DeepSeek: генерация контента, история запросов.
5. **lib.sh** – **общая библиотека** (логирование, создание файлов, запрос переменных, прогресс-бар), подключается всеми скриптами.

**Принцип модульности и гибкий порядок установки:**  
- Скрипт `init.sh` должен быть выполнен первым в любом сценарии.
- Скрипт `cms.sh` выполняется после `init.sh` и полностью настраивает сайт и CMS за один запуск.
- Скрипты `vpn.sh` и `ai.sh` выполняются после `cms.sh` (зависят от наличия домена и админки).
- Каждый скрипт перед началом работы проверяет наличие `lib.sh` в той же директории. Если файл отсутствует – выводит сообщение об ошибке и завершается.

### 1.3. Требования к окружению
- **Операционная система**: Ubuntu 24.04 LTS (рекомендовано) или Debian 12.
- **Права доступа**: скрипты запускаются от `root`.
- **Идемпотентность**: повторный запуск не должен нарушать работу системы. Скрипты проверяют наличие уже установленных компонентов и конфигураций, при обнаружении существующих объектов пропускают шаги (если не указан флаг `--force`).

### 1.4. Формат вывода информации
- В консоль выводится **название текущего шага** в формате `[X%] Действие`.
- Все подробные логи команд направляются в файл `/var/log/setup.log` с временными метками.
- По окончании скрипт выводит итоговое сообщение, содержащее:
  - Перечень выполненных действий.
  - Важные параметры (сгенерированные пароли, URL для доступа, пути к файлам).
  - Рекомендации по дальнейшим шагам.

**Единый алгоритм шагов:**  
- В начале скрипта определяется общее количество шагов.
- Каждый шаг нумеруется, его название выводится в консоль через функцию `show_progress`.
- После выполнения ключевых действий (например, после настройки nginx) обязательно выполняется перезапуск соответствующих сервисов с проверкой статуса.

### 1.5. Конфигурационный файл `.env`
Все изменяемые параметры задаются через файл `.env`, расположенный в одной директории со скриптами. Скрипты проверяют наличие файла и необходимых ключей; при отсутствии – запрашивают ввод в интерактивном режиме и сохраняют в `.env`.  
При проверке переменных используется конструкция `${!var-}` (с дефисом), чтобы избежать ошибки `unbound variable` при включённом режиме `set -u`.

**Обязательные переменные** (в зависимости от скрипта):
- `DOMAIN` – доменное имя сайта.
- `ADMIN_EMAIL` – email для уведомлений и сертификатов.
- `DB_NAME`, `DB_USER`, `DB_PASSWORD` – параметры базы данных сайта.
- `ADMIN_PASSWORD` – пароль администратора CMS (генерируется, если не задан).
- `DB_ROOT_PASSWORD` – пароль root MariaDB (генерируется в `init.sh`).
- `SSL_TYPE` – тип SSL-сертификата (`letsencrypt` или `existing`), а также пути к сертификатам (для `existing`).

### 1.6. Единая библиотека `lib.sh`
Все скрипты подключают `lib.sh`, который содержит:

- **Функции логирования:** `log_only`, `log`, `run_cmd`.
- **Функции прогресса:** `show_progress`.
- **Функции создания файлов:** `create_php_file` (с проверкой синтаксиса и резервным копированием).
- **Функции запроса переменных:** `ask_var` (с валидацией, поддержкой значений по умолчанию).
- **Функции загрузки `.env`:** `load_env` (с проверкой обязательных ключей).
- **Функции работы с БД:** `table_exists`, `column_exists`.
- **Общие переменные:** `RED`, `GREEN`, `YELLOW`, `NC`, `LOG_FILE`.

**Проверка наличия `lib.sh`:**  
Каждый скрипт в самом начале (после `set -euo pipefail`) выполняет:
```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
if [[ ! -f "$SCRIPT_DIR/lib.sh" ]]; then
    echo -e "${RED}Ошибка: файл lib.sh не найден в директории $SCRIPT_DIR.${NC}"
    exit 1
fi
source "$SCRIPT_DIR/lib.sh"
```

Благодаря `lib.sh` обеспечивается **единообразный дизайн** всех скриптов: одинаковый внешний вид прогресса, сообщений об ошибках, обработка флага `--force`.

## 2. Скрипт 1: Инициализация сервера (`init.sh`)

### 2.1. Цель
Подготовка чистой системы Ubuntu 24.04: обновление, установка базового ПО, настройка безопасности, установка выбранных пользователем компонентов (веб-стек, VPN, файловое хранилище и т.д.), подготовка инфраструктуры для резервного копирования.

### 2.2. Предварительные условия
- Чистая установка Ubuntu 24.04 (или Debian 12) с доступом в интернет.
- Скрипт запускается от root.

### 2.3. Последовательность действий

#### 2.3.1. Проверка прав и наличие `lib.sh`
Скрипт проверяет, что он выполняется с правами суперпользователя (root). Если нет – выводит сообщение об ошибке и завершается. Также проверяет наличие `lib.sh` в своей директории. Создаётся или очищается лог-файл `/var/log/setup.log`. В лог записывается строка о начале выполнения скрипта.

#### 2.3.2. Загрузка/создание `.env`
Если файл `.env` отсутствует, он создаётся пустым. Затем скрипт загружает переменные из `.env` в окружение с помощью `set -a; source .env; set +a`. Если переменная `INSTALLED_PURPOSES` существует, она разбирается в массив уже установленных целей.

#### 2.3.3. Интерактивный выбор целей установки
Пользователю показывается список доступных целей:
1. Сайт (веб-стек: Nginx + PHP + MariaDB)
2. Файловое хранилище (FTP/SFTP)
3. Сервер 3x-ui (прокси-панель)
4. 3x-ui совместно с сайтом (комбинация 1+3)
5. База данных (только MariaDB)
6. Базовое ПО (минимальный набор)

Если ранее уже были установлены какие-либо цели, они отображаются как уже установленные и не предлагаются повторно. Пользователь вводит номера целей через запятую (например, `1,3`). Можно выбрать несколько целей. Если выбрана цель 4 (совместная установка), она автоматически включает в себя цели 1 и 3, но в `.env` сохраняется именно 4 (для логики повторного запуска). Ввод `0` приводит к выходу из скрипта без изменений. После выбора формируется список уникальных целей, которые будут установлены.

#### 2.3.4. Обновление системы
Выполняется неинтерактивное обновление списков пакетов (`apt update`) и установка всех доступных обновлений (`apt upgrade -y`). Используются переменные окружения `DEBIAN_FRONTEND=noninteractive` и опции `-o Dpkg::Options::="--force-confdef" -o Dpkg::Options::="--force-confold"` для автоматического согласия с изменениями конфигурационных файлов.

#### 2.3.5. Установка базового ПО (обязательно для всех целей)
Устанавливаются следующие пакеты, если они ещё не установлены: `curl`, `wget`, `git`, `ufw`, `fail2ban`, `cron`, `openssl`, `python3`, `unattended-upgrades`, `software-properties-common`, `apt-transport-https`, `ca-certificates`, `gnupg`, `lsb-release`. Проверка осуществляется через `dpkg -l`.

#### 2.3.6. Настройка автоматических обновлений безопасности
Создаётся конфигурационный файл `/etc/apt/apt.conf.d/50unattended-upgrades`, который разрешает обновления только из репозитория безопасности. Служба `unattended-upgrades` включается и запускается. Если в `.env` задан `ADMIN_EMAIL`, он добавляется в качестве получателя уведомлений.

#### 2.3.7. Установка выбранных компонентов

**Если выбрана цель 1 (сайт) или цель 4 (сайт+VPN) или цель 5 (только БД):**
- **Установка MariaDB:** устанавливаются пакеты `mariadb-server` и `mariadb-client`. Если пароль `DB_ROOT_PASSWORD` ещё не задан в `.env`, генерируется случайный пароль (20 символов, буквы и цифры). Выполняется безопасная настройка, аналогичная `mysql_secure_installation`: установка пароля для root, удаление анонимных пользователей, запрет удалённого входа для root, удаление тестовой базы данных. Пароль сохраняется в `/root/.my.cnf` с правами 600.
- **Если выбрана цель 1 или 4 (сайт):** дополнительно устанавливается PHP 8.3 и необходимые расширения: `php8.3-fpm`, `php8.3-mysql`, `php8.3-curl`, `php8.3-xml`, `php8.3-mbstring`, `php8.3-zip`, `php8.3-gd`, `php8.3-intl`, `php8.3-bcmath`. Настраивается пул PHP-FPM (`/etc/php/8.3/fpm/pool.d/www.conf`) с параметрами: `pm = dynamic`, `pm.max_children = 20`, `pm.start_servers = 5`, `pm.min_spare_servers = 3`, `pm.max_spare_servers = 10`. PHP-FPM включается и перезапускается. Также устанавливается Nginx из официального репозитория. В глобальной конфигурации Nginx отключается вывод версии (`server_tokens off`). Создаётся шаблон конфигурации виртуального хоста для будущего сайта. Nginx включается и перезапускается. Дополнительно устанавливаются Composer (глобально) и Node.js (версия 20.x).

**Если выбрана цель 2 (файловое хранилище):**
- Устанавливается `vsftpd`. Создаётся пользователь `filestore` (если не существует) со случайным паролем. Настраивается базовая конфигурация vsftpd: разрешена локальная аутентификация, запись, chroot для пользователя. Сервис включается и перезапускается. В UFW открывается порт 21.

**Если выбрана цель 3 (сервер 3x-ui) или цель 4 (сайт+VPN):**
- Устанавливаются зависимости: `sqlite3`, `jq`, `socat`. Создаётся пользователь `xui` (если не существует). Создаётся директория `/opt/3x-ui`. Сама панель 3X-UI на этом этапе не устанавливается – это будет сделано позже скриптом `vpn.sh`.

#### 2.3.8. Настройка файервола (UFW)
Устанавливаются правила по умолчанию: запретить входящие, разрешить исходящие. Разрешаются порты: `22/tcp` (SSH), `80/tcp` (HTTP), `443/tcp` (HTTPS). Если выбрана цель 2, дополнительно разрешается `21/tcp` (FTP). UFW включается принудительно (`ufw --force enable`).

#### 2.3.9. Настройка fail2ban
Создаётся файл `/etc/fail2ban/jail.local` с включёнными джейлами для SSH (`[sshd]`) и для Nginx (`[nginx-http-auth]`). Служба fail2ban включается и перезапускается.

#### 2.3.10. Системные лимиты и резервное копирование
В файл `/etc/sysctl.conf` добавляется строка `fs.file-max = 65535`, применяется `sysctl -p`. Создаётся директория `/var/backups` с правами 700. Создаётся шаблон скрипта резервного копирования `/usr/local/bin/backup.sh` (содержит примеры команд для бэкапа БД и файлов сайта). Скрипт делается исполняемым. В cron (файл `/etc/crontab`) добавляется задание на ежедневный запуск бэкапа в 2:00 ночи.

#### 2.3.11. Сохранение выбранных целей в `.env` и завершение
Переменная `INSTALLED_PURPOSES` обновляется в `.env` (список исходных номеров целей, выбранных пользователем, без расширения зависимостей). Также в `.env` сохраняются все сгенерированные пароли (например, `DB_ROOT_PASSWORD`). Выводится итоговая информация: версии установленного ПО, созданные пользователи, пути к важным файлам. Даются рекомендации по дальнейшим шагам (запуск `cms.sh` для установки CMS и настройки домена).

## 3. Скрипт 2: Установка CMS и настройка сайта (`cms.sh`)

### 3.1. Цель
**Объединённый скрипт**, который выполняет полную настройку сайта и CMS:
- Запрос домена, email, выбор типа SSL-сертификата.
- Для выбранного типа `existing` – конвертация предоставленных файлов (приватный ключ, сертификат домена, CA-цепочка) в формат `fullchain.pem` и `privkey.pem` и размещение в `/etc/letsencrypt/live/${DOMAIN}/`.
- Настройка виртуального хоста Nginx.
- Получение или установка SSL-сертификата.
- Создание базы данных и пользователя.
- Установка ядра CMS: таблицы БД, админ-панель, роутер, трекер посещений, TinyMCE.
- Настройка cron для сбора метрик сервера.
- Добавление в админ-панель функционала для автоматического внедрения кода аналитики в существующие HTML/PHP-файлы (кнопка «Встроить аналитику»).
- Вывод инструкции, куда зайти в админке и что нажать для подключения аналитики к уже размещённым файлам сайта.

### 3.2. Предварительные условия
- Выполнен `init.sh` (установлен веб-стек).
- Скрипт запускается от root.

### 3.3. Последовательность действий

#### 3.3.1. Проверка окружения и загрузка `.env`
Скрипт подключает `lib.sh`, проверяет наличие файла. Загружает `.env`. Если переменная `DOMAIN` отсутствует, она запрашивается у пользователя (с валидацией доменного имени). Если отсутствует `ADMIN_EMAIL`, также запрашивается (с валидацией email). Все введённые значения сохраняются в `.env`.

#### 3.3.2. Выбор типа SSL-сертификата и конфигурация
Скрипт запрашивает у пользователя: использовать новый сертификат Let’s Encrypt или уже существующий из локальной папки.

**Для Let’s Encrypt:** дополнительно запрашивается `ADMIN_EMAIL` (если ещё не задан). В дальнейшем `certbot` получит сертификат автоматически.

**Для существующего сертификата:** скрипт запрашивает путь к папке, где лежат файлы:
- Файл с расширением `.key` (приватный ключ).
- Файл с расширением `.crt` или `.cer` (сертификат домена).
- Опционально – файл CA-цепочки (`ca.crt`, `ca.cer` или любой файл, содержащий промежуточные сертификаты).

Скрипт проверяет наличие этих файлов. Если какой-то обязательный файл не найден – выводится ошибка с просьбой указать корректный путь. Далее:
- Создаётся директория `/etc/letsencrypt/live/${DOMAIN}/` (если не существует).
- Файл сертификата домена и CA-цепочка (если есть) объединяются в `fullchain.pem` (сначала сертификат домена, затем CA-цепочка). Если CA-цепочки нет, `fullchain.pem` становится копией сертификата домена.
- Приватный ключ копируется в `privkey.pem`.
- Устанавливаются права: `fullchain.pem` – 644, `privkey.pem` – 600.
- В `.env` сохраняются `SSL_TYPE=existing`, `SSL_CERT_PATH=/etc/letsencrypt/live/${DOMAIN}/fullchain.pem`, `SSL_KEY_PATH=/etc/letsencrypt/live/${DOMAIN}/privkey.pem`.

#### 3.3.3. Создание корневой директории сайта
Создаётся директория `/var/www/${DOMAIN}` (если не существует). Внутри неё создаются поддиректории: `core`, `admin`, `templates`, `uploads`. Для `uploads` устанавливаются права 750 и владелец `www-data`.

#### 3.3.4. Создание `config.php`
Создаётся файл `/var/www/${DOMAIN}/config.php` (если его нет). Шаблон содержит константы `SITE_NAME` (значение по умолчанию «Мой сайт»), `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASS`, `DEBUG_MODE`. Также в шаблоне реализована автозагрузка классов из папки `core` и инициализация PDO-соединения с сохранением объекта в константу `DB_CONNECTION` и глобальную переменную `$pdo`.

#### 3.3.5. Создание базы данных и пользователя
Скрипт подключается к MariaDB (с использованием `/root/.my.cnf` для root-доступа). Выполняются запросы:
- `CREATE DATABASE IF NOT EXISTS \`${DB_NAME}\` CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;`
- `CREATE USER IF NOT EXISTS '${DB_USER}'@'localhost' IDENTIFIED BY '${DB_PASSWORD}';`
- `GRANT ALL PRIVILEGES ON \`${DB_NAME}\`.* TO '${DB_USER}'@'localhost';`
- `FLUSH PRIVILEGES;`

Если переменные `DB_NAME`, `DB_USER`, `DB_PASSWORD` не были заданы в `.env`, они генерируются:
- `DB_NAME` = преобразование домена (точки заменяются на подчёркивания).
- `DB_USER` = `user_${DB_NAME}`.
- `DB_PASSWORD` = случайная строка (20 символов, буквы и цифры).
Сгенерированные значения сохраняются в `.env`.

#### 3.3.6. Создание таблиц базы данных (CMS)
Скрипт выполняет SQL-запросы с конструкцией `CREATE TABLE IF NOT EXISTS` для всех необходимых таблиц:
- `users` – id, login, password_hash, role, email, created_at.
- `pages` – id, slug, title, content, meta_description, status, template, created_at, updated_at.
- `settings` – key (PRIMARY KEY), value.
- `visits` – id, visit_date, visitor_ip, user_agent, page_url, created_at, INDEX(visit_date).
- `server_stats` – id, recorded_at, load_1min, load_5min, load_15min, memory_total, memory_used, disk_total, disk_used, INDEX(recorded_at).
- `sessions` – session_id (PRIMARY KEY), data, last_updated.
- `files` – id, original_name, path, size, type, uploaded_at, uploaded_by (FOREIGN KEY to users.id).

Если таблица `users` пуста, добавляется запись администратора: логин `admin`, пароль из переменной `ADMIN_PASSWORD` (хэшируется с помощью `password_hash`), роль `admin`, email из `ADMIN_EMAIL`. Если `ADMIN_PASSWORD` не задан, генерируется случайный пароль и сохраняется в `.env`.

#### 3.3.7. Создание файлов административной панели
С использованием функции `create_php_file` создаются все PHP-файлы админки (полный перечень из 20+ файлов: `functions.php`, `auth.php`, `login.php`, `logout.php`, `index.php` (дашборд), API для графиков, `stats.php`, `visitors.php`, `settings.php`, `users.php`, `content.php`, `edit_page.php`, `delete_page.php`, `edit_file.php`, `import_to_db.php`, `upload.php`, `files.php`, `file-list.php`, `file-picker.html`, а также `header.php`, `sidebar.php`, `content_functions.php`). Также создаются языковые файлы `ru.php` и `en.php` с полным набором ключей локализации (более 70 ключей).

#### 3.3.8. Создание CSS-файла
Создаётся `/var/www/${DOMAIN}/admin/css/admin.css` с поддержкой светлой и тёмной темы, стилями для карточек, таблиц, кнопок.

#### 3.3.9. Установка TinyMCE
Скрипт проверяет наличие `npm`. При отсутствии – автоматически устанавливает Node.js и npm. Затем устанавливает TinyMCE 6.8.3 через `npm` и копирует в `/var/www/${DOMAIN}/admin/tinymce/`.

#### 3.3.10. Создание роутера `cms-router.php`
Создаётся файл `/var/www/${DOMAIN}/cms-router.php`, который реализует маршрутизацию: сначала поиск страницы в БД (таблица `pages` со статусом `published`), затем отдача статического файла, иначе 404.

#### 3.3.11. Автоматизация внедрения аналитики через админку
В административной панели создаётся раздел (или кнопка на дашборде) «Встроить аналитику». При нажатии:
- Открывается интерфейс, показывающий список HTML/PHP-файлов в корне сайта (кроме `admin`, `uploads`, `core`, `templates`, `tinymce`).
- Для каждого файла предлагается действие: «Добавить трекер» или «Удалить трекер».
- При добавлении трекера:
  - Для PHP-файлов: скрипт вставляет код трекера после первого `<?php` (или в начало файла, если открывающего тега нет).
  - Для HTML-файлов: скрипт добавляет `<script src="/js/tracker.js"></script>` перед `</body>`.
- Все изменения сохраняются с созданием резервной копии.

Код трекера (PHP-версия):
```php
if (isset($pdo) && strpos($_SERVER["REQUEST_URI"], "/admin") !== 0) {
    try {
        $ip = $_SERVER["REMOTE_ADDR"] ?? "";
        $ua = $_SERVER["HTTP_USER_AGENT"] ?? "";
        $url = $_SERVER["REQUEST_URI"] ?? "";
        $stmt = $pdo->prepare("INSERT INTO visits (visit_date, visitor_ip, user_agent, page_url) VALUES (CURDATE(), ?, ?, ?)");
        $stmt->execute([$ip, $ua, $url]);
    } catch (Exception $e) {}
}
```

JS-трекер (файл `/js/tracker.js`):
```javascript
(function() {
    if (window._trackerInited) return;
    window._trackerInited = true;
    function trackVisit() {
        const url = window.location.pathname + window.location.search;
        const data = new URLSearchParams();
        data.append("url", url);
        navigator.sendBeacon("/track.php", data);
    }
    if (document.readyState === "complete") {
        trackVisit();
    } else {
        window.addEventListener("load", trackVisit);
    }
})();
```

А также создаётся эндпоинт `/track.php`, который обрабатывает POST-запросы от JS-трекера и записывает визит в БД.

#### 3.3.12. Настройка виртуального хоста Nginx
Создаётся конфигурация `/etc/nginx/sites-available/${DOMAIN}`:
- Если выбран Let’s Encrypt или существующий сертификат – сразу создаётся конфигурация с HTTPS (прослушивание портов 80 и 443, редирект с HTTP на HTTPS). Если сертификат ещё не получен (например, для Let’s Encrypt на этом этапе), создаётся временная конфигурация только на порту 80.
- Затем запускается `certbot` (для Let’s Encrypt) или копируются существующие сертификаты (для existing) в `/etc/letsencrypt/live/${DOMAIN}/` (с конвертацией, как описано в 3.3.2).
- После получения сертификата конфигурация обновляется: добавляются директивы `ssl_certificate`, `ssl_certificate_key`, и блок `server` для порта 80 с редиректом.
- Активация сайта (симлинк в `sites-enabled`), проверка конфигурации, перезапуск Nginx.

#### 3.3.13. Настройка cron для сбора метрик сервера
Создаётся скрипт `/usr/local/bin/collect_server_stats.sh`, который каждые 5 минут собирает load average, память, диск и вставляет в `server_stats`, а также удаляет старые записи согласно `stats_retention`. Добавляется задание в `/etc/crontab`.

#### 3.3.14. Установка прав доступа
Все файлы в `/var/www/${DOMAIN}` получают владельца `www-data:www-data`. Директориям – 755, файлам – 644. `uploads` – 750, `.env` – 600.

#### 3.3.15. Перезапуск служб и завершение
Выполняется перезапуск PHP-FPM и Nginx. Выводится итоговая информация:
- URL сайта (https://domain).
- URL админ-панели (https://domain/admin).
- Логин `admin` и пароль из `.env`.
- Параметры базы данных.
- **Инструкция по подключению аналитики:**  
  «Для автоматического добавления кода сбора статистики на ваши существующие страницы (HTML/PHP) зайдите в административную панель → раздел «Аналитика» (или на дашборде нажмите кнопку «Встроить аналитику»). Выберите нужные файлы и нажмите «Добавить трекер». Система сама внедрит необходимый код и создаст резервные копии.»
- Рекомендация по запуску `vpn.sh` и `ai.sh` (опционально).

## 4. Скрипт 3: Развертывание VPN-сервера (`vpn.sh`)

### 4.1. Цель
Интеграция VPN-сервера (3X-UI) с уже работающим сайтом на одном домене через порт 443. Реализуется схема TLS-терминации на стороне 3X-UI с пробросом обычного HTTPS-трафика на локальный веб-сервер (fallback).

### 4.2. Предварительные условия
- Выполнен `init.sh` (установлены зависимости 3x-ui).
- Выполнен `cms.sh` (настроен домен, получен SSL-сертификат).
- В `.env` заданы `DOMAIN` и `ADMIN_EMAIL`.

### 4.3. Полная последовательность действий

#### 4.3.1. Проверка окружения и загрузка `.env`
Скрипт подключает `lib.sh`, проверяет наличие файла. Загружает `.env`. Проверяет наличие переменных `DOMAIN`, `ADMIN_EMAIL`. Если какой-то переменной нет – выводит ошибку и завершается.

#### 4.3.2. Проверка наличия SSL-сертификата
Скрипт проверяет существование файлов `/etc/letsencrypt/live/${DOMAIN}/fullchain.pem` и `/etc/letsencrypt/live/${DOMAIN}/privkey.pem`. Если они отсутствуют, запускается `certbot --nginx -d ${DOMAIN} --non-interactive --agree-tos --email ${ADMIN_EMAIL}` для получения сертификата. Если получение не удалось – скрипт завершается с ошибкой.

#### 4.3.3. Определение свободного локального порта для Nginx
Скрипт проверяет порты, начиная с 10443, пока не найдёт свободный (не используется в `ss -tlnp`). Выбранный порт сохраняется в переменную `NGINX_LOCAL_PORT` и в `.env` (для возможного использования при повторном запуске).

#### 4.3.4. Модификация конфигурации Nginx
Скрипт создаёт резервную копию текущего файла конфигурации `/etc/nginx/sites-available/${DOMAIN}`. Затем изменяет в нём:
- Удаляет все директивы `listen`, кроме `listen 127.0.0.1:${NGINX_LOCAL_PORT} ssl;` и `listen [::]:...` (если есть).
- Убеждается, что пути к SSL-сертификатам остаются корректными.
- Проверяет конфигурацию (`nginx -t`). Если проверка успешна – перезапускает Nginx. В случае ошибки – восстанавливает конфигурацию из резервной копии и завершается с ошибкой.

#### 4.3.5. Установка 3X-UI (если не установлен)
Скрипт проверяет наличие файла `/usr/local/x-ui/bin/xray-linux-amd64`. Если его нет, выполняет:
```bash
yes | bash <(curl -Ls https://raw.githubusercontent.com/mhsanaei/3x-ui/master/install.sh)
```
В процессе установки автоматически соглашается на все вопросы. После установки панель останавливается командой `x-ui stop`.

#### 4.3.6. Генерация параметров панели управления
Скрипт генерирует случайные значения:
- `XUI_PORT` – число от 52000 до 53000.
- `XUI_PATH` – случайная строка из 16 символов (буквы и цифры), начинающаяся с `/` (например, `/aB3dEfGhIjKlMnOp`).
- `XUI_USERNAME` – случайная строка из 12 символов.
- `XUI_PASSWORD` – случайная строка из 20 символов.
Эти значения сохраняются в `.env`.

#### 4.3.7. Настройка панели через утилиту `x-ui`
Выполняются команды:
```bash
x-ui setting -port ${XUI_PORT} -username ${XUI_USERNAME} -password ${XUI_PASSWORD}
```
Для установки пути к панели (`webBasePath`) требуется прямое редактирование базы данных SQLite:
```bash
sqlite3 /etc/x-ui/x-ui.db "UPDATE settings SET value='${XUI_PATH}' WHERE key='webBasePath';"
```

#### 4.3.8. Настройка UFW для доступа к панели
Скрипт проверяет переменную `ADMIN_IP` в `.env`. Если она задана, разрешает доступ к порту `${XUI_PORT}` только с этого IP: `ufw allow from ${ADMIN_IP} to any port ${XUI_PORT} proto tcp`. Если `ADMIN_IP` не задана, выводит предупреждение и открывает порт для всех: `ufw allow ${XUI_PORT}/tcp`. Также открываются порты `443/tcp` и `443/udp` (если ещё не открыты). UFW перезагружается.

#### 4.3.9. Создание inbound для VPN
Скрипт генерирует ключи Reality:
```bash
PRIVATE_KEY=$(/usr/local/x-ui/bin/xray-linux-amd64 x25519)
PUBLIC_KEY=$(echo "$PRIVATE_KEY" | grep "Public key:" | awk '{print $3}')
PRIVATE_KEY=$(echo "$PRIVATE_KEY" | grep "Private key:" | awk '{print $3}')
```
Формирует JSON-конфигурацию inbound:
```json
{
  "protocol": "vless",
  "port": 443,
  "settings": {
    "clients": [],
    "decryption": "none",
    "fallbacks": [{"dest": "127.0.0.1:${NGINX_LOCAL_PORT}", "xver": 0}]
  },
  "streamSettings": {
    "network": "tcp",
    "security": "reality",
    "realitySettings": {
      "show": false,
      "dest": "${DOMAIN}:443",
      "xver": 0,
      "serverNames": ["${DOMAIN}"],
      "privateKey": "${PRIVATE_KEY}",
      "publicKey": "${PUBLIC_KEY}",
      "minClientVer": "",
      "maxClientVer": "",
      "maxTimeDiff": 0,
      "shortIds": ["", "1234567890abcdef"]
    }
  },
  "sniffing": {
    "enabled": true,
    "destOverride": ["http", "tls"]
  }
}
```
Этот JSON вставляется в таблицу `inbounds` базы данных 3X-UI через `sqlite3`.

#### 4.3.10. Создание первого клиента
Генерируется UUID (через `cat /proc/sys/kernel/random/uuid`). Формируется JSON клиента:
```json
{
  "id": "сгенерированный-uuid",
  "flow": "xtls-rprx-vision",
  "email": "user1@${DOMAIN}",
  "limitIp": 0,
  "totalGB": 0,
  "expiryTime": 0,
  "enable": true,
  "tgId": "",
  "subId": ""
}
```
Клиент добавляется к созданному inbound через прямое обновление JSON в базе данных (с использованием `sqlite3` и утилиты `jq`).

#### 4.3.11. Перезапуск служб и завершение
Выполняется `systemctl restart x-ui` (или `x-ui restart`). Nginx уже был перезапущен ранее. Проверяется доступность сайта (curl к https://domain) и доступность панели (curl к https://domain:${XUI_PORT}${XUI_PATH}/login). Выводится итоговая информация:
- URL панели управления: `https://${DOMAIN}:${XUI_PORT}${XUI_PATH}`
- Логин: `${XUI_USERNAME}`
- Пароль: `${XUI_PASSWORD}`
- Параметры VPN-клиента:
  - Адрес: `${DOMAIN}`
  - Порт: 443
  - UUID: сгенерированный
  - Flow: `xtls-rprx-vision`
  - SNI: `${DOMAIN}`
  - PublicKey: `${PUBLIC_KEY}`
  - ShortId: (один из сгенерированных)
- Порт для подписок (определяется автоматически из базы 3X-UI, обычно 2096).
- Рекомендации по настройке клиентов (например, импорт конфигурации через QR-код, который можно сгенерировать в веб-интерфейсе панели).

## 5. Скрипт 4: Интеграция AI DeepSeek (`ai.sh`)

### 5.1. Цель
Добавление в админку модуля генерации контента через API DeepSeek, сохранение истории запросов, интеграция с редактором TinyMCE.

### 5.2. Предварительные условия
- Выполнен `cms.sh` (необходима таблица `settings` и ядро админки).

### 5.3. Полная последовательность действий

#### 5.3.1. Проверка окружения и загрузка `.env`
Скрипт подключает `lib.sh`, проверяет наличие файла. Загружает `.env`. Проверяет наличие переменной `DOMAIN`. Если её нет – завершается с ошибкой.

#### 5.3.2. Добавление настроек API в админку
Скрипт добавляет в файл `/var/www/${DOMAIN}/admin/settings.php` новый блок настроек:
- Поле для ввода `API Key DeepSeek` (тип `password`).
- Поле для выбора модели (по умолчанию `deepseek-chat`).
- Поле для задания максимальной длины ответа (токенов).
При сохранении эти параметры записываются в таблицу `settings` с ключами `deepseek_api_key`, `deepseek_model`, `deepseek_max_tokens`.

#### 5.3.3. Создание таблицы истории запросов
Выполняется SQL-запрос:
```sql
CREATE TABLE IF NOT EXISTS ai_requests (
    id INT AUTO_INCREMENT PRIMARY KEY,
    prompt TEXT NOT NULL,
    response TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    user_id INT NOT NULL,
    INDEX (user_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 5.3.4. Создание страницы генерации контента
Создаётся файл `/var/www/${DOMAIN}/admin/ai_generator.php` со следующим функционалом:
- Интерфейс: текстовое поле для промпта, кнопка «Отправить», область вывода результата.
- При отправке POST-запроса:
  - Получает API-ключ из настроек.
  - Формирует запрос к `https://api.deepseek.com/v1/chat/completions` (JSON: модель, сообщения, температура и т.д.).
  - Отправляет запрос с заголовком `Authorization: Bearer ${API_KEY}`.
  - Получает ответ, сохраняет в таблицу `ai_requests` (вместе с ID текущего пользователя).
  - Выводит ответ в формате HTML с сохранением разметки.
- Под ответом добавляются кнопки:
  - «Создать страницу» – перенаправляет на `/admin/edit_page.php?prefill_title=...&prefill_content=...` (передаёт сгенерированный текст).
  - «Вставить в редактор» – если открыт редактор TinyMCE, вызывает JavaScript для вставки текста.

#### 5.3.5. Создание раздела «История запросов AI»
Создаётся файл `/var/www/${DOMAIN}/admin/ai_history.php`:
- Таблица со списком запросов (пагинация по 20 записей).
- Колонки: дата, промпт (обрезанный до 100 символов), ответ (обрезанный до 100 символов), действия (просмотр полной версии).
- При клике на «просмотр» открывается модальное окно с полным промптом и ответом.
- Возможность удалить запись (только для администратора).

#### 5.3.6. Интеграция с TinyMCE
Скрипт модифицирует файл `/var/www/${DOMAIN}/admin/edit_page.php` (и, возможно, `edit_file.php` для HTML-файлов):
- Добавляет в панель инструментов TinyMCE новую кнопку «AI».
- При нажатии открывается модальное окно (создаётся отдельный HTML-блок), содержащее поле для промпта.
- После отправки запроса к API (через AJAX на `/admin/ai_api.php`) ответ вставляется в текущее положение курсора в редакторе.

#### 5.3.7. Создание API-эндпоинта для AJAX
Создаётся файл `/var/www/${DOMAIN}/admin/ai_api.php`:
- Принимает POST-запрос с параметром `prompt`.
- Проверяет авторизацию пользователя (должен быть залогинен).
- Вызывает API DeepSeek (аналогично п. 5.3.4).
- Сохраняет запрос в историю.
- Возвращает JSON с полем `response`.

#### 5.3.8. Добавление пункта меню
Скрипт обновляет `/var/www/${DOMAIN}/admin/includes/sidebar.php`, добавляя пункт меню:
```php
<li class="nav-item">
    <a class="nav-link <?= $current_page=="ai_generator.php"?"active":"" ?>" href="/admin/ai_generator.php">
        <i class="bi bi-robot"></i> <?= __('ai_generator') ?>
    </a>
</li>
```
Также добавляется пункт для истории (только для администратора).

#### 5.3.9. Установка прав доступа и перезапуск служб
Все созданные файлы получают права 644, владельца `www-data`. Дополнительный перезапуск Nginx не требуется, так как изменения только в PHP-файлах.

#### 5.3.10. Вывод итоговой информации
Скрипт выводит:
- Сообщение об успешной интеграции DeepSeek.
- Ссылки на новые разделы админки: «Генератор контента» (`/admin/ai_generator.php`) и «История AI» (`/admin/ai_history.php`).
- Рекомендацию: «Для работы с AI необходимо добавить API-ключ DeepSeek в разделе Настройки → AI Settings.»

## 6. Общая библиотека `lib.sh` (детальная спецификация)

### 6.1. Переменные
```bash
RED='\033[0;31m'; GREEN='\033[0;32m'; YELLOW='\033[1;33m'; NC='\033[0m'
LOG_FILE="/var/log/setup.log"
FORCE_MODE=false   # устанавливается из аргументов скрипта
```

### 6.2. Функции

| Функция | Описание | Пример вызова |
|---------|----------|---------------|
| `log_only "сообщение"` | Пишет в лог-файл с временной меткой. | `log_only "Начало установки"` |
| `log "сообщение"`   | Выводит сообщение в консоль (с цветом) и пишет в лог. | `log "${GREEN}Готово${NC}"` |
| `run_cmd команда`   | Выполняет команду, логирует её, stdout/stderr направляет в лог. | `run_cmd systemctl restart nginx` |
| `show_progress текущий общий "текст"` | Выводит процент выполнения. | `show_progress 2 10 "Настройка БД"` |
| `create_php_file файл "содержимое"` | Создаёт PHP-файл с проверкой синтаксиса, бэкапом при `--force`. | `create_php_file "$file" "<?php ..."` |
| `ask_var имя_переменной "подсказка" значение_по_умолчанию функция_валидации` | Интерактивный запрос с дефолтом. Валидация – имя функции, возвращающей 0/1. | `ask_var DOMAIN "Введите домен" "example.com" validate_domain` |
| `load_env [обязательные_переменные...]` | Загружает `.env`, проверяет наличие указанных переменных, при отсутствии завершает с ошибкой. | `load_env "DOMAIN" "ADMIN_EMAIL"` |
| `table_exists таблица` | Проверяет существование таблицы в БД (использует `mysql` с опциями из `.env`). | `table_exists "users"` |
| `column_exists таблица колонка` | Проверяет существование колонки. | `column_exists "pages" "status"` |
| `is_pkg_installed пакет` | Проверяет, установлен ли deb-пакет. | `is_pkg_installed "nginx"` |

### 6.3. Проверка наличия `lib.sh` в каждом скрипте
В начале каждого скрипта (после `set -euo pipefail`) обязательно добавляется:
```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
if [[ ! -f "$SCRIPT_DIR/lib.sh" ]]; then
    echo -e "${RED}Ошибка: файл lib.sh не найден в директории $SCRIPT_DIR.${NC}"
    exit 1
fi
source "$SCRIPT_DIR/lib.sh"
```

## 7. Хранение чувствительных данных

- **`.env`** – основной источник для скриптов. Права 600, владелец root. Содержит:
  - `DOMAIN`, `ADMIN_EMAIL`
  - `DB_NAME`, `DB_USER`, `DB_PASSWORD`
  - `ADMIN_PASSWORD`
  - `DB_ROOT_PASSWORD` (только для `init.sh`)
  - `SSL_TYPE`, `SSL_CERT_PATH`, `SSL_KEY_PATH` (после `cms.sh`)
  - Параметры VPN (после `vpn.sh`)
  - Ключ API DeepSeek (после `ai.sh`)
- **`config.php`** – для PHP-приложения. Содержит те же параметры БД, что и `.env` (копия). Находится в корне сайта, но его содержимое не видно через веб (интерпретируется PHP). Безопасно.
- **База данных** – хранит настройки сайта (название, тема, срок хранения статистики), содержимое страниц, историю AI-запросов. Пароли не хранит.

**Вывод:** три места хранения – это нормально, если соблюдены права доступа и нет дублирования секретов между БД и файлами.

## 9. Заключение

Предложенная архитектура с единой библиотекой `lib.sh` и объединённым скриптом `cms.sh` обеспечивает:

- **Минимальное количество скриптов** – всего 4 основных + библиотека.
- **Простоту использования** – после `init.sh` достаточно запустить один `cms.sh`, чтобы получить полностью работающий сайт с CMS и админкой.
- **Гибкость работы с SSL** – поддержка как Let’s Encrypt, так и готовых сертификатов с автоматической конвертацией.
- **Автоматизацию внедрения аналитики** – через удобный интерфейс админ-панели, без ручного редактирования файлов.
- **Единообразие** – все скрипты используют одинаковые функции вывода, логирования, создания файлов.
- **Безопасность** – правильное хранение секретов, проверка прав.
