# Разработка комплекса скриптов автоматизированного развертывания сервера, CMS, VPN и интеграции с AI (DeepSeek)

## Версия 9.1 (Итоговая, с детальным описанием функциональности)

---

## 1. Общие положения

### 1.1. Цель проекта
Создать набор последовательных и модульных скриптов для автоматической установки и настройки сервера под управлением Linux, развертывания на нём сайта с собственной CMS, добавления VPN-сервера, работающего на том же домене, и последующей интеграции модуля генерации контента на основе AI DeepSeek. Все скрипты должны выполняться на «чистой» системе с минимальным участием оператора и обеспечивать получение полностью работоспособного продукта.

### 1.2. Общая концепция и модульность («кубики»)
Проект состоит из шести скриптов:

1. **init.sh** – базовая подготовка сервера (обязательный для всех).
2. **site.sh** – создание структуры сайта, настройка веб-сервера и базы данных.
3. **cms_core.sh** – ядро CMS: система пользователей, дашборд, статистика, настройки.
4. **cms_content.sh** – управление контентом: страницы, визуальный редактор, загрузка файлов.
5. **vpn.sh** – развертывание VPN-сервера (3X-UI) с интеграцией на одном домене.
6. **ai.sh** – интеграция с AI DeepSeek: генерация контента, история запросов.

**Принцип модульности:**  
- Скрипт `init.sh` должен быть выполнен первым в любом сценарии.  
- Скрипты со 2-го по 6-й могут выполняться независимо друг от друга **при условии**, что все необходимые для их работы компоненты уже присутствуют на сервере.  
- Каждый скрипт перед началом работы должен проверять наличие критических зависимостей. При отсутствии – выдавать понятное сообщение об ошибке и завершаться.

### 1.3. Требования к окружению
- **Операционная система**: Ubuntu 24.04 LTS (рекомендовано) или Debian 12.
- **Права доступа**: скрипты запускаются от `root`.
- **Идемпотентность**: повторный запуск не должен нарушать работу системы. Скрипты проверяют наличие уже установленных компонентов и конфигураций, при обнаружении существующих объектов пропускают шаги (если не указан флаг `--force`).

### 1.4. Формат вывода информации
- В консоль выводится **название текущего шага** (например, «Создание структуры директорий сайта»).
- Все подробные логи команд направляются в файл `/var/log/setup.log` с временными метками.
- По окончании скрипт выводит итоговое сообщение, содержащее:
  - Перечень выполненных действий.
  - Важные параметры (сгенерированные пароли, URL для доступа, пути к файлам).
  - Рекомендации по дальнейшим шагам.

**Единый алгоритм шагов:**  
- В начале скрипта определяется общее количество шагов.
- Каждый шаг нумеруется, его название выводится в консоль.
- После выполнения ключевых действий (например, после настройки nginx) обязательно выполняется перезапуск соответствующих сервисов с проверкой статуса.

### 1.5. Конфигурационный файл `.env`
Все изменяемые параметры задаются через файл `.env`, расположенный в одной директории со скриптами. Скрипты проверяют наличие файла и необходимых ключей; при отсутствии – запрашивают ввод в интерактивном режиме и сохраняют в `.env`.  
При проверке переменных используется конструкция `${!var-}` (с дефисом), чтобы избежать ошибки `unbound variable` при включённом режиме `set -u`.

## 2. Скрипт 1: Инициализация сервера (`init.sh`)

### 2.1. Цель
Подготовка чистой системы Ubuntu 24.04: обновление, установка базового ПО, создание пользователя, настройка безопасности, установка выбранных пользователем компонентов (веб-стек, VPN, файловое хранилище и т.д.), подготовка инфраструктуры для резервного копирования.

### 2.2. Предварительные условия
- Чистая установка Ubuntu 24.04 (или Debian 12) с доступом в интернет.
- Скрипт запускается от root.

### 2.3. Последовательность действий

1. **Проверка прав и инициализация лога**  
   - Проверка, что скрипт выполняется от root.
   - Создание/очистка лог-файла `/var/log/setup.log`.

2. **Загрузка/создание конфигурационного файла `.env`**  
   - Если файл `.env` отсутствует, он создаётся пустым.
   - Загружаются переменные из `.env`. Если переменная `INSTALLED_PURPOSES` существует, она разбирается в массив уже установленных целей.

3. **Интерактивный выбор целей установки**  
   - Пользователю показывается список доступных целей:
     1. Сайт (веб-стек: Nginx + PHP + MariaDB)
     2. Файловое хранилище (FTP/SFTP)
     3. Сервер 3x-ui (прокси-панель)
     4. 3x-ui совместно с сайтом (комбинация 1+3)
     5. База данных (только MariaDB)
     6. Базовое ПО (минимальный набор)
   - Если ранее уже были установлены какие-либо цели, они отображаются как уже установленные и не предлагаются повторно.
   - Пользователь вводит номера целей через запятую (например, `1,3`). Можно выбрать несколько целей.
   - Если выбрана цель 4 (совместная установка), она автоматически включает в себя цели 1 и 3, но в `.env` сохраняется именно 4 (для логики повторного запуска).
   - Ввод `0` приводит к выходу из скрипта без изменений.
   - После выбора формируется список уникальных целей, которые будут установлены.

4. **Запрос дополнительных параметров**  
   - В зависимости от выбранных целей запрашиваются необходимые параметры:
     - Если выбрана цель 1 (сайт) или 4 (сайт+VPN), запрашивается `ADMIN_EMAIL` (для сертификатов и уведомлений) и опционально публичный SSH-ключ для пользователя `webmaster`.
     - Если выбрана цель 2 (файловое хранилище), запрашиваются параметры FTP/SFTP (пользователь, пароль, директория).
     - Если выбрана цель 3 или 4, запрашиваются параметры для панели 3x-ui (позже они будут сгенерированы автоматически).
   - Все введённые данные сохраняются в `.env`.

5. **Обновление системы**  
   - Выполняется неинтерактивное обновление списков пакетов и установка всех доступных обновлений.

6. **Установка базового ПО (обязательно для всех целей)**  
   - Устанавливаются пакеты: `mc`, `curl`, `wget`, `git`, `unzip`, `zip`, `software-properties-common`, `apt-transport-https`, `ca-certificates`, `gnupg`, `lsb-release`, `cron`, `sqlite3`, `openssl`, `python3`, `certbot`, `python3-certbot-nginx`, `ufw`, `fail2ban`.  
   - Проверка через `dpkg -l`, установка только недостающих.

7. **Создание пользователя `webmaster` и настройка SSH**  
   - Создаётся пользователь `webmaster` (если не существует) с домашней директорией и добавлением в группу `sudo`.
   - Если в `.env` задан `PUBLIC_KEY`, он добавляется в `~webmaster/.ssh/authorized_keys`.
   - В конфигурации SSH отключается вход по паролю для root (`PermitRootLogin prohibit-password`).
   - **Важно:** после изменения конфигурации выполняется перезапуск службы SSH. В Ubuntu 24.04 служба называется `ssh`, а не `sshd`. Скрипт должен использовать корректное имя службы.

8. **Настройка автоматических обновлений безопасности**  
   - Устанавливается пакет `unattended-upgrades`.
   - Создаётся конфигурационный файл, разрешающий обновления только из репозитория безопасности.
   - В качестве получателя уведомлений указывается `ADMIN_EMAIL` (если задан).
   - Служба включается и запускается.

9. **Настройка файервола (UFW) и fail2ban**  
   - В UFW разрешаются порты 22 (SSH), 80 (HTTP), 443 (HTTPS).  
   - UFW включается принудительно.
   - Для fail2ban создаётся базовая конфигурация (`jail.local`) с защитой SSH и nginx (включается джейл `nginx-http-auth`).  
   - Служба fail2ban включается и перезапускается.

10. **Установка и настройка выбранных компонентов (в зависимости от целей)**  

    **Если выбрана цель 1 (сайт) или 4 (сайт+VPN) или 5 (только БД):**
    - **Установка и настройка MariaDB**  
      - Устанавливаются `mariadb-server` и `mariadb-client`.
      - Генерируется случайный пароль для root (если он ещё не задан в `.env`).
      - Выполняются операции, аналогичные `mysql_secure_installation`:
        - Установка пароля для root.
        - Удаление анонимных пользователей.
        - Запрет удалённого входа для root.
        - Удаление тестовой базы данных.
      - Пароль root сохраняется в `/root/.my.cnf` с правами 600.
    - **Установка PHP и расширений** (если выбрана цель 1 или 4)  
      - Устанавливается PHP 8.3 и необходимые расширения: `php8.3-fpm`, `php8.3-mysql`, `php8.3-curl`, `php8.3-json`, `php8.3-xml`, `php8.3-mbstring`, `php8.3-zip`, `php8.3-gd`, `php8.3-intl`, `php8.3-bcmath`.
      - Настраивается пул PHP-FPM (`www.conf`) для работы с объёмом оперативной памяти 1 ГБ:
        - `pm = dynamic`
        - `pm.max_children = 20`
        - `pm.start_servers = 5`
        - `pm.min_spare_servers = 3`
        - `pm.max_spare_servers = 10`
      - PHP-FPM включается и перезапускается.
    - **Установка и настройка Nginx** (если выбрана цель 1 или 4)  
      - Устанавливается Nginx из официального репозитория.
      - В глобальной конфигурации отключается вывод версии (`server_tokens off`).
      - Создаётся шаблон конфигурации виртуального хоста для будущего сайта.
      - Nginx включается и перезапускается.

    **Если выбрана цель 3 (сервер 3x-ui) или 4 (сайт+VPN):**
    - Установка 3X-UI будет выполнена позже скриптом `vpn.sh`. На этом этапе только подготавливается окружение (уже установлены MariaDB, Nginx, PHP).

    **Если выбрана цель 2 (файловое хранилище):**
    - Устанавливается и настраивается FTP/SFTP-сервер (например, vsftpd или openssh-sftp-server). Параметры доступа берутся из `.env` или запрашиваются.

11. **Установка дополнительных инструментов (общих для всех целей)**  
    - Устанавливается Composer глобально.
    - Устанавливается Node.js (LTS версия 20.x).
    - В системных параметрах увеличивается лимит открытых файлов (`fs.file-max = 65535`).

12. **Настройка резервного копирования**  
    - Создаётся директория `/var/backups/site` с правами 700.
    - Создаётся шаблон скрипта резервного копирования `/usr/local/bin/backup.sh` с примерами команд для будущего использования (бэкап БД и файлов сайта).
    - В cron добавляется задание на ежедневный запуск этого скрипта в 2:00.

13. **Завершение и сохранение выбранных целей**  
    - Выбранные цели записываются в `.env` в переменную `INSTALLED_PURPOSES` (список номеров через запятую, без расширения зависимостей).
    - Выполняется перезапуск всех затронутых сервисов: SSH, Nginx (если установлен), fail2ban, PHP-FPM (если установлен).
    - Выводится сводная информация о версиях установленного ПО, созданных пользователях, путях к важным файлам.
    - Даются рекомендации по дальнейшим действиям (запуск `site.sh` для сайта, `vpn.sh` для VPN и т.д.).

## 3. Скрипт 2: Установка сайта (`site.sh`)

### 3.1. Цель
Создание структуры сайта, настройка виртуального хоста Nginx, получение SSL-сертификата, инициализация базы данных и таблиц.

### 3.2. Предварительные условия
- Выполнен `init.sh` с выбором цели «сайт».
- Наличие интернета для получения SSL-сертификата.

### 3.3. Последовательность действий

1. **Проверка окружения и загрузка конфигурации**  
   - Проверка, что скрипт выполняется от root.
   - Загрузка переменных из файла `.env`. Если файл отсутствует – ошибка.
   - Проверка наличия обязательных переменных (`DOMAIN`, `ADMIN_EMAIL`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`).  
     Если какая-либо переменная отсутствует, она запрашивается у пользователя интерактивно. Для `DB_PASSWORD` при отсутствии генерируется случайный пароль.
   - Все введённые значения сохраняются в `.env`.
   - Если `ADMIN_PASSWORD` не задан, генерируется случайный пароль и также сохраняется в `.env`.

2. **Создание структуры директорий сайта**  
   - Создаётся корневая директория `/var/www/<DOMAIN>` (где `<DOMAIN>` — значение из `.env`).
   - Внутри неё создаются поддиректории: `core`, `admin`, `templates`, `uploads`.
   - Если директория уже существует и не пуста, скрипт выводит предупреждение и пропускает создание (если не указан флаг `--force`).
   - Устанавливаются права доступа: `755` для `core`, `admin`, `templates`; `750` для `uploads`.

3. **Создание файлов `index.php` и `config.php`**  
   - Создаётся файл `index.php` в корне сайта. Этот файл служит единой точкой входа и содержит базовую маршрутизацию (заглушку).
   - Создаётся файл `config.php`, содержащий константы подключения к базе данных (из `.env`), автозагрузку классов из папки `core` и инициализацию PDO-соединения.
   - Если файлы уже существуют, они не перезаписываются (если не указан `--force`).

4. **Настройка виртуального хоста Nginx**  
   - Создаётся конфигурационный файл `/etc/nginx/sites-available/<DOMAIN>` со следующими настройками:
     - Прослушивание порта 80.
     - Корневая директория `/var/www/<DOMAIN>`.
     - Обработка PHP-файлов через сокет PHP-FPM (версия определяется автоматически).
     - Директива `try_files`, направляющая запросы к `index.php` при отсутствии файла.
     - Защита директории `uploads` от выполнения PHP-скриптов.
     - Отключение вывода версии сервера (`server_tokens off`).
   - Конфигурация активируется созданием символической ссылки в `sites-enabled`.
   - Выполняется проверка конфигурации (`nginx -t`). В случае успеха Nginx перезагружается.

5. **Получение SSL-сертификата (Let’s Encrypt)**  
   - Запускается `certbot` с плагином nginx, в неинтерактивном режиме, с автоматическим соглашением с условиями.
   - В качестве домена и email используются значения из `.env`.
   - После успешного получения сертификата пути к сертификату и ключу сохраняются в `.env` (переменные `SSL_CERT_PATH` и `SSL_KEY_PATH`).

6. **Инициализация базы данных и создание таблиц**  
   - Создаётся база данных с именем `DB_NAME`, если она ещё не существует (кодировка `utf8mb4`).
   - Создаётся пользователь `DB_USER` с паролем `DB_PASSWORD`, если он ещё не существует.
   - Пользователю выдаются **все привилегии** на созданную базу данных.
   - Создаются таблицы (если их нет):
     - `users` – для хранения учётных записей администраторов и редакторов.
     - `pages` – для хранения страниц сайта (slug, заголовок, контент, мета-описание, статус).
     - `settings` – для хранения ключей и значений настроек.
   - Добавляется запись администратора: логин `admin`, пароль (хэшируется с помощью `openssl passwd -6`), роль `admin`, email из `ADMIN_EMAIL`.
   - Если таблица `pages` пуста, добавляется главная страница по умолчанию.

7. **Завершение**  
   - Выполняется перезапуск Nginx и PHP-FPM.
   - Выводится итоговая информация: URL сайта (http или https в зависимости от наличия сертификата), логин и пароль администратора, пути к конфигурационным файлам и сертификатам.
   - Даются рекомендации по защите файла `.env` и переходу к следующему этапу (установке CMS).

## 4. Скрипт 3: Ядро CMS и базовая админка (`cms_core.sh`)

### 4.1. Цель
Создание административной панели: система аутентификации и управления пользователями, дашборд с реальной информацией о сервере и сайте, страницы статистики, настройки, а также сбор метрик сервера и посещений.

### 4.2. Предварительные условия
- Выполнен `site.sh`.
- Таблицы `users`, `pages`, `settings` существуют.

### 4.3. Последовательность действий

1. **Создание дополнительных таблиц**  
   - Создаётся таблица `server_stats` для хранения метрик сервера (загрузка CPU, использование памяти и диска, временная метка).
   - Опционально создаётся таблица `sessions` для хранения сессий в базе данных (если выбран такой способ хранения).

2. **Создание структуры административной панели**  
   - В папке `/var/www/<DOMAIN>/admin` создаются поддиректории: `includes`, `css`, `js`, `api`.

3. **Создание файла аутентификации (`auth.php`)**  
   - Файл размещается в `admin/includes/`.
   - Содержит функции для проверки авторизации (`isLoggedIn`), требования входа (`requireLogin`), проверки роли администратора (`requireAdmin`), получения текущего пользователя (`currentUser`).

4. **Создание страницы входа (`login.php`)**  
   - Форма для ввода логина и пароля.
   - При отправке проверяет наличие пользователя в таблице `users` и сверяет пароль через `password_verify`.
   - При успехе устанавливает сессию и перенаправляет на главную страницу админки.

5. **Создание страницы выхода (`logout.php`)**  
   - Уничтожает сессию и перенаправляет на страницу входа.

6. **Создание дашборда (`index.php`)**  
   - Отображает карточки с текущей загрузкой CPU, использованием RAM, диска, количеством опубликованных страниц, посещениями за сегодня.
   - Показывает график посещаемости за последние 7 дней (данные получаются через API).
   - Отображает таблицу последних 5 посещений.

7. **Создание API для графика посещаемости**  
   - Эндпоинт `/admin/api/visits_last_7.php` возвращает JSON с датами и количеством посещений за последние 7 дней для построения графика Chart.js.

8. **Создание общего шаблона (header, sidebar)**  
   - Файлы `header.php` и `sidebar.php` в `admin/includes/`.
   - В шапке отображается имя текущего пользователя, в боковом меню – ссылки на все разделы админки с проверкой прав доступа.

9. **Создание страницы управления пользователями (`users.php`)**  
   - Доступна только администратору.
   - Отображает список существующих пользователей с возможностью удаления (кроме собственного).
   - Форма для добавления нового пользователя: логин, пароль, роль (editor/viewer), email.
   - При добавлении пароль хэшируется.

10. **Создание страницы статистики сервера (`stats.php`)**  
    - Отображает графики загрузки CPU, использования RAM и диска за последние 24 часа (данные из таблицы `server_stats`).
    - Данные для графиков получаются через API `/admin/api/server_stats.php`.

11. **Создание страницы посетителей (`visitors.php`)**  
    - Таблица с записями о посещениях (IP, страница, user agent, время).
    - Фильтры по дате и IP.
    - Отображает общее количество посещений и уникальных IP за выбранный период.

12. **Создание страницы настроек (`settings.php`)**  
    - Позволяет редактировать параметры из таблицы `settings`: название сайта, email администратора, тему админки (светлая/тёмная), срок хранения статистики, настройки SMTP (опционально).
    - При сохранении обновляет соответствующие записи в таблице.

**Дополнительные действия:**
- В корневой `index.php` (фронтенд сайта) добавляется код для записи посещений в таблицу `visits` (при каждом обращении к сайту).
- Создаётся cron-задание (каждые 5 минут), которое запускает скрипт сбора метрик сервера. Скрипт собирает загрузку CPU, использование памяти и диска и вставляет их в таблицу `server_stats`.

**Завершение:** перезапуск Nginx и PHP-FPM, вывод сообщения о готовности админки.

## 5. Скрипт 4: Управление контентом и файлами (`cms_content.sh`)

### 5.1. Цель
Добавление в админку разделов для управления страницами (CRUD) и загрузки файлов, интеграция с визуальным редактором TinyMCE.

### 5.2. Предварительные условия
- Выполнен `cms_core.sh` (или как минимум существуют таблицы `pages` и `users`).

### 5.3. Последовательность действий

1. **Установка и настройка визуального редактора TinyMCE**  
   - Скачивается архив TinyMCE версии 6.8.3 (последняя MIT-версия) и распаковывается в `/var/www/<DOMAIN>/admin/tinymce/`.
   - В шаблоне редактирования страницы подключается скрипт редактора и выполняется инициализация с параметрами:
     - `selector` — идентификатор текстового поля.
     - `images_upload_url` — адрес эндпоинта для загрузки изображений.
     - `file_picker_callback` — функция для выбора ранее загруженных файлов.
     - `content_css` — опциональный путь к пользовательскому CSS для единообразия стиля.

2. **Создание страницы управления страницами**  
   - Список всех страниц с возможностью фильтрации по статусу и поиска.
   - Для каждой страницы доступны действия: редактирование, удаление, дублирование.
   - Форма создания/редактирования страницы содержит поля:
     - Заголовок.
     - Slug (генерируется из заголовка, редактируемый, проверка уникальности).
     - Контент (визуальный редактор).
     - Мета-описание.
     - Статус (черновик/опубликовано).
     - Шаблон (выбор из файлов в папке `templates`).
   - При сохранении страницы данные записываются в таблицу `pages`.

3. **Создание раздела загрузки файлов**  
   - Страница со списком загруженных файлов (таблица: оригинальное имя, путь, размер, дата, действия).
   - Интерфейс загрузки (drag-and-drop, множественная) с проверкой MIME-типов и размера (настройки из таблицы `settings`).
   - Эндпоинт `/admin/upload.php` обрабатывает загрузку:
     - Проверяет авторизацию.
     - Проверяет MIME-тип (с помощью `finfo_file`) и размер.
     - Сохраняет файл в папку `uploads/YYYY/MM/` с безопасным именем (случайная строка).
     - Добавляет запись в таблицу `files` (оригинальное имя, путь, размер, тип, ID пользователя, время).
     - Возвращает JSON с ключом `location` (URL файла).
   - Эндпоинт `/admin/file-list` возвращает JSON со списком файлов (для выбора в `file_picker_callback`).

4. **Доработка фронтенда сайта**  
   - В корневом `index.php` реализуется маршрутизация:
     - Из URL извлекается slug.
     - Если существует статический файл (например, `.html`), он отдаётся напрямую.
     - Иначе ищется запись в таблице `pages` с таким slug и статусом `published`.
     - Если страница найдена, подключается выбранный шаблон из папки `templates` и отображается контент.
     - Если страница не найдена — возвращается ответ 404.

**Завершение:** перезапуск Nginx и PHP-FPM, вывод сообщения о добавленных возможностях.

## 6. Скрипт 5: Развертывание VPN-сервера (`vpn.sh`)

### 6.1. Цель
Интеграция VPN-сервера (3X-UI) с уже работающим сайтом на одном домене через порт 443. Реализуется схема TLS-терминации на стороне 3X-UI с пробросом обычного HTTPS-трафика на локальный веб-сервер (fallback).

### 6.2. Предварительные условия
- Выполнен `init.sh` и `site.sh` (или вручную настроенный сайт с SSL).
- В `.env` заданы `DOMAIN` и `ADMIN_EMAIL`.

### 6.3. Последовательность действий

1. **Проверка окружения и загрузка конфигурации**  
   - Проверка, что скрипт выполняется от root.
   - Загрузка переменных из `.env`. При отсутствии `DOMAIN` или `ADMIN_EMAIL` – интерактивный запрос.
   - Проверка наличия SSL-сертификата для `DOMAIN` в `/etc/letsencrypt/live/$DOMAIN/`.
   - Если сертификат отсутствует, временно настраивается Nginx на порт 80, запускается certbot для получения сертификата, затем Nginx возвращается к исходной конфигурации.

2. **Настройка Nginx для fallback**  
   - Определяется свободный локальный порт (начиная с 10443).
   - Модифицируется конфигурация Nginx для домена: директива `listen` меняется на `127.0.0.1:<порт> ssl;` (прослушивание только локального интерфейса).
   - Все другие директивы `listen` удаляются.
   - Убеждаемся, что пути к SSL-сертификатам указаны корректно.
   - Проверяется конфигурация и перезапускается Nginx.

3. **Установка 3X-UI (неинтерактивно)**  
   - Проверяется наличие установленной 3X-UI по наличию файла `/usr/local/x-ui/bin/xray-linux-amd64`.
   - Если 3X-UI не установлен, запускается официальный скрипт установки с автоматическим подтверждением всех вопросов (через `yes`).
   - После установки панель останавливается для настройки.

4. **Генерация и применение параметров панели**  
   - Генерируются случайные значения:
     - Порт для веб-интерфейса (в диапазоне 52000–53000).
     - Путь к панели (случайная строка, начинающаяся с `/`).
     - Логин и пароль администратора панели.
   - Параметры сохраняются в `.env`.
   - Через утилиту `x-ui setting` устанавливаются порт, логин и пароль.
   - Через прямое редактирование базы SQLite (`/etc/x-ui/x-ui.db`) устанавливается путь к панели (`webBasePath`).
   - Настраивается UFW: открывается порт панели. Если в `.env` задан `ADMIN_IP`, доступ разрешается только с этого IP, иначе – для всех с предупреждением.

5. **Создание inbound для VPN**  
   - Генерируются ключи Reality с помощью `xray x25519`.
   - Формируется JSON-конфигурация inbound с протоколом `vless`, портом `443`, сетью `tcp`, безопасностью `reality`.
   - В настройках указывается `dest: "$DOMAIN:443"`, `serverNames: ["$DOMAIN"]`.
   - Задаётся fallback: `{"dest": 127.0.0.1:<локальный_порт>}` (порт, на который теперь слушает Nginx).
   - JSON вставляется в таблицу `inbounds` базы данных 3X-UI через `sqlite3`.
   - Создаётся первый клиент с UUID и параметром `flow: xtls-rprx-vision`.
   - В UFW открываются порты `443/tcp` и `443/udp`, а также порт сервиса подписок (по умолчанию 2096, определяется из базы).

6. **Завершение**  
   - Перезапускаются Nginx и Xray.
   - Выполняется проверка доступности сайта и панели.
   - Выводится итоговая информация: URL сайта, URL панели управления (с портом и путём), логин/пароль панели, параметры первого VPN-клиента (адрес, порт, UUID, flow, SNI), порт подписок.
   - Даются рекомендации по безопасности (ограничить доступ к панели, сменить пароль, настроить fail2ban).

### 6.4. Важные технические нюансы (учитываются в скрипте)
- **Fallback должен указывать IP:порт** – обязательно `127.0.0.1:10443`, а не только порт.
- **Локальный порт Nginx слушает plain HTTP** – без `ssl`, так как расшифровку уже выполнил Xray.
- **ALPN в настройках TLS inbound** – `http/1.1` для корректной работы с Nginx, ожидающим HTTP/1.1.
- **Flow** задаётся для каждого клиента, а не для inbound.
- **Установка 3X-UI** выполняется с `yes |`, чтобы автоматически ответить на все вопросы.
- **Пути к SSL-сертификатам** после получения сохраняются в `.env` для использования панелью.

## 7. Скрипт 6: Интеграция AI DeepSeek (`ai.sh`)

### 7.1. Цель
Добавление в админку модуля генерации контента через API DeepSeek, сохранение истории запросов, интеграция с редактором TinyMCE.

### 7.2. Предварительные условия
- Выполнены `site.sh` и `cms_core.sh` (необходима таблица `settings` и ядро админки).

### 7.3. Последовательность действий

1. **Добавление настроек API**  
   - В разделе «Настройки» административной панели появляется поле для ввода ключа API DeepSeek.
   - Ключ сохраняется в таблице `settings` (рекомендуется хранить в зашифрованном виде).

2. **Создание раздела «Генерация контента»**  
   - Создаётся страница `/admin/ai_generator.php` (или аналогичная).
   - Интерфейс содержит текстовое поле для ввода промпта и кнопку «Отправить».
   - При отправке скрипт обращается к API DeepSeek (POST-запрос с JSON, ключ в заголовке).
   - Ответ API отображается в форматированном виде.
   - Под ответом появляются кнопки:
     - «Создать страницу» – открывает форму предварительного заполнения полей страницы (заголовок, контент) с возможностью редактирования и сохранения в таблицу `pages`.
     - «Добавить в меню» (опционально) – после создания страницы предложить добавить ссылку в меню.

3. **Создание истории запросов**  
   - Создаётся таблица `ai_requests` с полями: `id`, `prompt`, `response`, `created_at`, `user_id`.
   - Каждый запрос к API сохраняется в эту таблицу.
   - В админке добавляется раздел «История запросов AI», где отображается список запросов с пагинацией и возможностью просмотра деталей.

4. **Интеграция с редактором TinyMCE**  
   - В редакторе TinyMCE добавляется кнопка «Сгенерировать с AI».
   - При нажатии открывается модальное окно с полем для промпта.
   - После отправки ответ AI вставляется в текущее положение курсора в редакторе.
   - При желании можно также предложить сразу создать новую страницу на основе сгенерированного текста.

5. **Дополнительные возможности (опционально, но рекомендуются)**  
   - **Генерация структуры страниц**: AI может предложить JSON со списком страниц (название, slug, примерное содержание). Скрипт разбирает JSON и создаёт несколько страниц одной кнопкой.
   - **Автоматическое обновление меню** при создании новых страниц (если в CMS реализовано управление меню).

**Завершение:** вывод сообщения о добавленных возможностях, ссылки на новые разделы админки.

## 8. Общие требования к разработке

### 8.1. Идемпотентность и проверки
- Перед созданием любых файлов или директорий скрипт проверяет их существование. При наличии – пропускает (если не указан флаг `--force`).
- Проверяет права доступа (например, директория `uploads` должна принадлежать `www-data` и иметь права `750`).
- После каждого скрипта перезапускаются затронутые сервисы (nginx, php-fpm, x-ui, fail2ban) и проверяется их статус.

### 8.2. Безопасность
- Все генерируемые пароли должны быть случайными (не менее 16 символов).
- Файл `.env` после завершения всех работ должен иметь права `600` и владельца `root`.
- Директория `uploads` защищена от выполнения PHP.
- Панель 3X-UI доступна только по случайному порту и пути, с возможностью ограничения по IP.
- MIME-тип загружаемых файлов проверяется через `finfo_file()`.
- Имена файлов на сервере генерируются случайно, исходное имя сохраняется в БД.

### 8.3. Логирование и отладка
- Весь подробный вывод команд направляется в `/var/log/setup.log` с временными метками.
- В консоль выводится только название текущего шага и итоговые сообщения.
- В лог попадают также события о пропуске шагов и ошибках прав.

### 8.4. Документация
- В начале каждого скрипта – комментарий с описанием назначения, зависимостей, версии.
- В комплекте – файл `.env.example` со всеми возможными переменными.

### 8.5. Тестирование
- Каждый скрипт тестируется на чистой Ubuntu 24.04.
- Проверяются сценарии: первый запуск, повторный запуск, запуск без выполнения предыдущих (с проверкой ошибок).

## 9. Известные проблемы и их решения (тонкие места)

| Проблема | Решение |
|----------|---------|
| **Установка 3X-UI зависает из-за интерактивных вопросов** | Использовать `yes \|` перед скриптом установки. |
| **Fallback не работает, сайт недоступен после настройки VPN** | В inbound указывать `"dest": "127.0.0.1:10443"`, а не только порт. |
| **Двойное шифрование (SSL + SSL)** | Nginx на локальном порту должен слушать plain HTTP, без `ssl`. |
| **Клиенты VPN не подключаются из-за неверного ALPN** | Установить ALPN = `http/1.1` в настройках TLS inbound. |
| **Пароль администратора не сохраняется в `.env`** | Генерировать и дописывать в `.env` на этапе `site.sh`. |
| **Права пользователя БД недостаточны** | Выдавать `GRANT ALL PRIVILEGES ON DB_NAME.* TO user@localhost`. |
| **После получения SSL пути не сохраняются для VPN** | После `certbot` записывать `SSL_CERT_PATH` и `SSL_KEY_PATH` в `.env`. |
| **Скрипты не перезапускают сервисы** | После каждого скрипта явно перезапускать nginx, php-fpm, x-ui, ufw. |
| **Пропуск шагов при повторном запуске работает некорректно** | Реализовать проверки существования файлов, конфигов и записей в БД. |
| **Ошибка `unbound variable` при проверке переменных в `.env`** | Использовать `${!var-}` вместо `${!var}`. |
| **Перезапуск SSH в Ubuntu 24.04** | Команда `systemctl restart ssh` (не `sshd`). |
